What Stage 1 does

Discovers candidate replication papers from two academic databases: OpenAlex and Semantic Scholar. OpenAlex is queried with 37 search phrases plus concept-based search (OpenAlex ML concepts for Replication and Reproducibility). Semantic Scholar uses the top 10 phrases. New results are appended to data/candidates.csv; already-seen papers are skipped via a persistent dedup index.

Every run also harvests all cached API responses from disk first — pages downloaded in any previous run flow into the CSV automatically without re-fetching.

--auto-advance processes one (source, phrase, year) combination per invocation, cycling through all phrases for all years. State is saved in cache/search_state.json and resumes across runs. Exit code 2 signals a full cycle completed with no new rows.

5-pass deduplication

Applied to each new batch before it is appended to candidates.csv. Runs in search/deduplicate.py.

PassWhat it removes
0a — DOI patternsfigshare datasets (10.6084/) and PeerJ peer-review objects (/reviews/ in DOI) — never scholarly papers
0b — Versioned preprintsLower-version preprint DOIs ending in _v1, _v2 etc. Keeps the canonical or highest-version DOI; drops all older versions
1 — Exact DOIDuplicate DOIs after clean_doi() normalisation; keeps the row with more non-empty metadata fields (richness score)
2 — Fuzzy titleDOI-less rows whose titles score ≥ 90 on RapidFuzz token_sort_ratio; keeps the richer row
3 — FLoRA cross-checkPapers already catalogued in data/flora_entry_sheet.csv (doi_r) and data/flora.csv (doi_r + doi_o) are dropped — no need to re-extract them

The persistent candidates index (cache/candidates_index.txt) stores up to 4 keys per row (OA ID → DOI → URL → title) so duplicates from any identifier are caught without loading the full CSV into memory. Built automatically on first run; ~1s to load thereafter.

What Stage 2 does

Removes false positives and classifies candidates into replication, reproduction, needs_review, or false_positive. Reads candidates.csv in 50 k-row chunks to avoid OOM on files with 1M+ rows.

The rule filter runs first and handles ~99% of rows with no API calls. Only rows marked needs_review by the rules are sent to the LLM. Results are streamed row-by-row to filtered.csv via a persistent index (cache/filtered_index.txt) so interrupted runs resume without reprocessing.

Decision logic
Condition (checked in order)Result
Title/abstract matches a biological/technical exclusion phrase (DNA, cell, viral, replication fork, replication stress, code replication, data replication…)false_positive
No include keyword phrase found in title or abstractfalse_positive
Include phrase found AND title/abstract contains an author-year citation pattern like "(Author, YYYY)"replication or reproduction
Include phrase found but no citation pattern — rule is uncertain→ LLM
LLM returns a definitive classificationreplication, reproduction, or false_positive
LLM cannot decideneeds_review
Replication phrases (32 patterns — any match passes Rule 2)

Defined in filter/phrase_detection.py. These are regex patterns applied to title + abstract. Pills show the human-readable form; actual matching uses word-boundary anchors and case-insensitive regex. Blue pills also trigger reproduction classification when no replication phrases co-occur.

replication of we replicated we replicate replicating the findings direct replication conceptual replication preregistered replication registered replication failed to replicate did not replicate could not reproduce successfully replicated reproducibility of replication and extension(s) registered report of close replication high-powered replication large-scale replication reproduced the findings attempted to replicate aimed to replicate set out to replicate successfully replicat* we conducted a replication many-labs replication multi-site replication replicating and extending replication study of study replicated our replication independent replication exact replication replication attempt cross-cultural replication cross-national replication cross-lab replication
LLM prompt (what the model receives)

For uncertain rows the LLM receives the paper title and abstract with this instruction:

Classify this paper as one of:
  replication   — directly replicates or attempts to reproduce a prior study
  reproduction  — reproduces methods in a different context, not testing the original claim
  false_positive — about replication in biology/CS, or unrelated

Title: {title}
Abstract: {abstract}

Respond with JSON: {"filter_status": "...", "filter_evidence": "...", "filter_confidence": "high|medium|low"}

LLM configured via FILTER_OPENAI_MODEL env var (primary); Gemini fallback via GEMINI_MODEL. Responses cached by hash(title + abstract) so identical rows never re-query the LLM.

python -m filter.run_filter
↓
Load + resume
Reads candidates.csv in 50k-row chunks (avoids OOM on 1M+ row files). Keys in cache/filtered_index.txt are skipped — run resumes from where it left off.
↓ per row
—
Rule 1 — Exclusion check
Title + abstract scanned for biological or technical patterns (DNA/RNA/viral/cell replication, replication fork/origin/stress, code/data/model replication…)
Match → false_positive [high confidence, rule_based] — stop here
↓ no match
—
Rule 2 — Replication phrase check
Any of 35+ replication phrases present in title or abstract?
None found → false_positive [high, rule_based] — stop here
↓ phrase found
Rule 3 — Type detection
Do only reproduction-flavoured phrases fire (could not reproduce, reproducibility of…)?
Yes → base_status = reproduction   No → base_status = replication
Not an exit — every surviving row continues to Rule 4.
↓
—
Rule 4 — Specific-target gate
Does text contain an author–year citation like (Author, YYYY) within a plausible year range?
No → uncertain — send to LLM   Yes → continue to the proximity gate
↓ cite found
—
Rule 5 — Same-sentence proximity gate
Is the citation in the same sentence as the replication phrase? A phrase and a citation that merely co-occur somewhere in the abstract are frequently unrelated.
No → uncertain — send to LLM
↓ same sentence
—
Rule pass — accepted by rules alone
Phrase and citation co-located → replication or reproduction [high, rule_based], no API call.
↓ uncertain rows from Rules 4 + 5
—

Title + abstract sent to GPT (FILTER_OPENAI_MODEL); Gemini fallback via GEMINI_MODEL. Response cached by hash(title+abstract) — identical rows never re-query.
Output: replication / reproduction / false_positive / needs_review with filter_evidence quote
↓
Write

What Stage 3 does

For each replication paper: (1) classifies the match type, (2) finds the original target study, (3) codes the replication outcome. Results are streamed to extracted.csv one row at a time.

DOI verification runs automatically on every written row — it fetches CrossRef metadata for doi_o and corrects hallucinated DOIs before they hit the CSV.

CLI
python -m extract.run_extract

# Options
--extracted-test   # write to extracted-test.csv (sandbox)
--resume           # skip already-processed DOIs
--no-llm           # rule-based only (faster, less accurate)
--doi 10.x/y       # process a single DOI

# DOI audit
python -m extract.audit_dois          # dry-run
python -m extract.audit_dois --apply  # write corrections

How it works — data flow

python -m extract.run_extract --resume
↓
Load + skip
Reads filtered.csv. Skips rows where filter_status = false_positive and DOIs already written to extracted.csv (or extracted-test.csv in sandbox mode).
↓ per replication / reproduction row
Step 1 — Classify match type
Rule check first — title patterns (Many Labs, RRR) or abstract count ≥ 3 → multiple_original immediately.
Otherwise LLM classifies: single_original multiple_match multiple_original
↓
multiple_original
Pipeline runs once per target study → N rows appended (original_rank 1, 2, 3…)
single / multiple_match
One pipeline run → one CSV row
↓
Step 2 — Link original study
1st Author-year match — (Author, YYYY) pattern in abstract → OpenAlex candidate lookup + disambiguation
2nd LLM abstract — abstract + title sent to LLM, which identifies the original study and returns a DOI guess
3rd LLM fulltext — PDF acquired via multi-tier waterfall (arXiv → OSF → Unpaywall → CORE…), then parsed by 6 methods; best-scoring parse fed to LLM
score = refs×300 + abstract_len + intro_len×2 + min(raw÷5, 1000)
no match target_pending — all methods failed; row flagged for manual review
↓ automatic on every row
Step 3 — DOI verification
CrossRef fetches real metadata for doi_o; title + year compared to what the LLM returned.
verified OK   corrected wrong DOI replaced   mismatch link_confidence → low
Replication's own doi_r always excluded as a correction target.
↓
Step 4 — Code outcome
Keyword rules run first (abstract + fulltext). LLM only fires when no keyword matches.
success failure mixed uninformative descriptive
↓
Step 5 — Write (streamed)
Row appended immediately to extracted.csv — Stage 4 validation tab can open while the pipeline is still running. Parquet cache refreshed.
What Extract-Test is

A sandbox for running and reviewing Stage 3 pipelines before committing results to extracted.csv. Writes to extracted-test.csv instead, and automatically skips DOIs already resolved in extracted.csv.

Use it to test new prompts, new models, or multi-original rows without risking the production file. Promote individual rows or everything when satisfied.

CLI
# Run into sandbox
python -m extract.run_extract --extracted-test
python -m extract.run_extract --extracted-test --resume

# Promote to production
python -m extract.promote_test --all           # promote all
python -m extract.promote_test --doi 10.x/y   # promote one
python -m extract.promote_test --all --dry-run # preview

What Stage 4 does

Human validators review each extracted row — checking the original DOI, replication type, and outcome. Votes are stored in Supabase (PostgreSQL). Two independent validators vote per row; disagreements are flagged for adjudication.

SUPABASE_SERVICE_KEY=eyJhbGc...

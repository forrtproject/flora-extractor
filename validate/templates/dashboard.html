What Stage 1 does

Discovers candidate replication papers from two academic databases: OpenAlex and Semantic Scholar. Each source is queried with 23 keyword phrases. New results are appended to data/candidates.csv; already-seen papers are skipped via a persistent dedup index.

Every run also harvests all cached API responses from disk first — pages downloaded in any previous run flow into the CSV automatically without re-fetching.

--auto-advance processes one (source, phrase, year) combination per invocation, cycling through all phrases for all years. State is saved in cache/search_state.json and resumes across runs. Exit code 2 signals a full cycle completed with no new rows.

5-pass deduplication

Applied to each new batch before it is appended to candidates.csv. Runs in search/deduplicate.py.

PassWhat it removes
0a — DOI patternsfigshare datasets (10.6084/) and PeerJ peer-review objects (/reviews/ in DOI) — never scholarly papers
0b — Versioned preprintsLower-version preprint DOIs ending in _v1, _v2 etc. Keeps the canonical or highest-version DOI; drops all older versions
1 — Exact DOIDuplicate DOIs after clean_doi() normalisation; keeps the row with more non-empty metadata fields (richness score)
2 — Fuzzy titleDOI-less rows whose titles score ≥ 90 on RapidFuzz token_sort_ratio; keeps the richer row
3 — FLoRA cross-checkPapers already catalogued in data/flora_entry_sheet.csv (doi_r) and data/flora.csv (doi_r + doi_o) are dropped — no need to re-extract them

The persistent candidates index (cache/candidates_index.txt) stores up to 4 keys per row (OA ID → DOI → URL → title) so duplicates from any identifier are caught without loading the full CSV into memory. Built automatically on first run; ~1s to load thereafter.

What Stage 2 does

Removes false positives and classifies candidates into replication, reproduction, needs_review, or false_positive. Reads candidates.csv in 50 k-row chunks to avoid OOM on files with 1M+ rows.

The rule filter runs first and handles ~99% of rows with no API calls. Only rows marked needs_review by the rules are sent to the LLM. Results are streamed row-by-row to filtered.csv via a persistent index (cache/filtered_index.txt) so interrupted runs resume without reprocessing.

Decision logic
Condition (checked in order)Result
Title/abstract matches a biological/technical exclusion phrase (DNA, cell, viral, replication fork, replication stress, code replication, data replication…)false_positive
No include keyword phrase found in title or abstractfalse_positive
Include phrase found AND title/abstract contains an author-year citation pattern like "(Author, YYYY)"replication or reproduction
Include phrase found but no citation pattern — rule is uncertain→ LLM
LLM returns a definitive classificationreplication, reproduction, or false_positive
LLM cannot decideneeds_review
LLM prompt (what the model receives)

For uncertain rows the LLM receives the paper title and abstract with this instruction:

Classify this paper as one of:
  replication   — directly replicates or attempts to reproduce a prior study
  reproduction  — reproduces methods in a different context, not testing the original claim
  false_positive — about replication in biology/CS, or unrelated

Title: {title}
Abstract: {abstract}

Respond with JSON: {"filter_status": "...", "filter_evidence": "...", "filter_confidence": "high|medium|low"}

LLM configured via FILTER_OPENAI_MODEL env var (primary); Gemini fallback via GEMINI_MODEL. Responses cached by hash(title + abstract) so identical rows never re-query the LLM.

What Stage 3 does

For each replication paper: (1) classifies the match type, (2) finds the original target study, (3) codes the replication outcome. Results are streamed to extracted.csv one row at a time.

DOI verification runs automatically on every written row — it fetches CrossRef metadata for doi_o and corrects hallucinated DOIs before they hit the CSV.

CLI
python -m extract.run_extract

# Options
--extracted-test   # write to extracted-test.csv (sandbox)
--resume           # skip already-processed DOIs
--no-llm           # rule-based only (faster, less accurate)
--doi 10.x/y       # process a single DOI

# DOI audit
python -m extract.audit_dois          # dry-run
python -m extract.audit_dois --apply  # write corrections

What Extract-Test is

A sandbox for running and reviewing Stage 3 pipelines before committing results to extracted.csv. Writes to extracted-test.csv instead, and automatically skips DOIs already resolved in extracted.csv.

Use it to test new prompts, new models, or multi-original rows without risking the production file. Promote individual rows or everything when satisfied.

CLI
# Run into sandbox
python -m extract.run_extract --extracted-test
python -m extract.run_extract --extracted-test --resume

# Promote to production
python -m extract.promote_test --all           # promote all
python -m extract.promote_test --doi 10.x/y   # promote one
python -m extract.promote_test --all --dry-run # preview

What Stage 4 does

Human validators review each extracted row — checking the original DOI, replication type, and outcome. Votes are stored in Supabase (PostgreSQL). Two independent validators vote per row; disagreements are flagged for adjudication.

Configuration
SUPABASE_URL=https://xxx.supabase.co
SUPABASE_SERVICE_KEY=eyJhbGc...

Add these to your .env file. Without them, this tab shows "not configured".

What this section covers

Gap analysis comparing what the current pipeline has extracted against the FLoRA entry sheet (107 known replications). Use it to find missed papers, filter misclassifications, and plan re-extraction.

All scripts are in analysis/. Data files live in analysis/ and are not committed to git.

CLI
python -m analysis.run_overlap_analysis  # gap analysis
python -m analysis.rule_analysis         # filter rule audit
python -m analysis.apa_resolver          # resolve APA citations
Analysis scripts
  • run_overlap_analysis.py Gap analysis — compares extracted.csv against FLoRA entry sheet. Writes gap CSVs and gap_summary.md.
  • rule_analysis.py Audits filter rules: contribution per rule, false-positive rate, improvement opportunities.
  • apa_resolver.py Resolves APA-format citations in extracted rows to structured DOIs via CrossRef.
Output files
  • gap_analysis_doi_matched.csvDOI-matched gap list
  • gap_analysis_url_matched.csvURL-matched gaps
  • gap_vs_extracted_comparison.csvSide-by-side comparison
  • rule_improvement_opportunities.csvActionable filter improvements
  • gap_summary.mdHuman-readable summary
